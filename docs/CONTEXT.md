# Project Context

Resume Analyzer is a portfolio project that compares a résumé against a job description and returns a concrete gap analysis: a "Semantic Match Score" (40% keyword match, 40% embedding cosine similarity, 20% experience/years heuristic), missing skills, and LLM-rewritten versions of weak resume bullets. It intentionally excludes auth, a database, background workers, and any AI features beyond the analysis itself — see `docs/PRD.md` and `docs/ARCHITECTURE.md`, which govern all implementation decisions (user instruction > PRD > architecture, per the root `CLAUDE.md`/`AGENTS.md`).

The backend (`backend/`, FastAPI, Python 3.9 venv at `backend/.venv`) is fully built. `POST /analyze` streams Server-Sent Events (`data: {"type","payload"}\n\n`) in this order: `status` messages during parsing/embedding, `result_partial` (score + skills, available even if the LLM path later fails), `status`/`bullet_result` pairs per weak bullet, `result_final` (the full `AnalysisResult`), then a final `status: "Done"`. Skill/keyword matching and experience extraction are pure heuristics (no LLM); Voyage AI provides embeddings for the similarity score. The LLM path (weak-bullet selection → prompt builder → provider → validator) supports Groq, Gemini, and OpenRouter via vendor SDKs behind a `LLMProvider` Protocol (`providers/llm/`, selected via `LLM_PROVIDER` in `.env`, currently `openrouter`). Every LLM rewrite passes through a spaCy-based hallucination guardrail (`services/validator.py`) that lemmatizes and diffs content words against the original resume text, flagging (never silently allowing) any rewrite that introduces unverifiable terms; a whitelist of known action verbs (`data/action_verbs.py`) avoids false positives on legitimate verb swaps (e.g. "Built" → "Engineered"). Malformed LLM JSON is retried once; guardrail failures are not retried, only flagged.

The frontend (`frontend/`, Next.js 16.2.10 App Router, React 19.2, Tailwind v4, TypeScript) is also fully built: a single-page flow (`app/page.tsx`) with an upload form for the résumé (PDF/txt, ≤5MB) and job description, consuming the SSE stream client-side via `fetch().body.getReader()` (not `EventSource`, since the request is a multipart POST). The UI follows a minimal/editorial style with a numbers-only score display, per explicit user design choices — no charts, bars, or animation. Components live in `app/components/` (`UploadForm`, `StatusBar`, `ScoreCard`, `SkillsPanel`, `BulletRewrites`); SSE parsing logic is isolated in `lib/analyze.ts` with types in `lib/types.ts` mirroring the backend's Pydantic models exactly. The score/skills panel stays rendered even if the LLM leg errors out later in the stream, matching the backend's "score path never depends on the LLM path" guarantee.

Both halves have been verified end-to-end together (real backend + real frontend, real API keys, `curl` and browser), not just unit-tested in isolation. Note: `AGENTS.md` at both the repo root and inside `frontend/` warns that this is a non-standard Next.js version (16) with real breaking changes from older training data (e.g. `middleware` renamed to `proxy`, async `params`/`searchParams`, Turbopack as default) — consult `frontend/node_modules/next/dist/docs/` before writing new frontend code rather than assuming familiar Next.js conventions.
